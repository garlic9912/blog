# 动态配置加载, 而非硬编码
| 硬编码                    | 动态加载                                     |
| ---------------------- | ---------------------------------------- |
| 将数据库地址、API密钥等配置直接写在代码里 | 配置存放在外部文件（`.env`,`yaml`）或环境变量中，程序运行时动态读取 |
|                        |                                          |
## ✅ 核心优势
#### 1. 🔒 安全性提升
- 敏感信息（API密钥、数据库密码）不再暴露在代码中
- 避免因代码泄露导致的核心资产损失
#### 2. 🔄 多环境无缝切换
- 同一份代码可在开发、测试、预发布、生产环境运行
- 无需修改源码，只需更改配置文件或环境变量
- 容器化部署（Docker/K8s）天然适配这种模式
#### 3. 🚀 运维灵活性
- 线上配置变更无需重新构建和部署应用
- 支持配置热更新（某些场景下可做到不重启服务）
- 统一配置中心（如Consul、Apollo）的接入基础
#### 4. 🧪 测试友好
- 单元测试可轻松覆盖各种配置组合
- 无需Mock大量硬编码常量


# 接口化设计
在项目初期，**不要过度设计**，但**要为未来的扩展预留接口**。先用 **PostgreSQL + Chroma** 作为默认实现快速跑通整个流程，同时在代码层面设计好抽象层，后续切换到其他数据库组件时无需重写核心代码。
## 关系型数据库接口
**设计思路**：不直接使用特定数据库驱动（如 `psycopg2`、`pymysql`），而是采用统一抽象层。
**扩展性体现**：
- 使用统一的数据库连接串（`DATABASE_URL`）
- 仅需修改配置文件中的连接串，即可无缝切换：
    - **PostgreSQL**：`postgresql://user:pass@localhost:5432/db
    - **MySQL**：`mysql+pymysql://user:pass@localhost:3306/db`
    - **SQLite**：`sqlite:///./db.sqlite3`
- 核心业务代码无需任何改动

## 向量数据库接口
**设计思路**：定义统一的接口规范，规定向量数据库必须实现的核心方法（如插入向量、相似度检索）。
**扩展性体现**：
- 定义一个 `BaseVectorStore` 抽象基类
- 所有向量数据库实现都必须继承并实现其定义的方法
- 系统启动时根据配置文件动态决定实例化哪个实现类
- 核心 Agent 代码只依赖于抽象接口，不依赖具体实现
