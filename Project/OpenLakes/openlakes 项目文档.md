# 项目背景
`OpenLakes` 是一个开源数据湖仓平台，现有组件包括 `MinIO`（对象存储）、`Trino`（查询引擎）、`Spark`, `Airflow`。 目前仓库中没有多模态查询能力，本模块目标是在 `OpenLakes` 上实现跨模态联合查询，支持图、时序、空间、向量、关系五种模态。 应用场景：河流污染溯源。给定下游传感器报警事件， 通过多模态联合查询找出上游可能的污染源并匹配相关法规。

# 设计选择
## 查询引擎的选择
目前 `OpenLakes` 集成了两种查询与计算引擎，分别为 `Spark` 和 `Trino`。其中，`Spark` 主要负责离线数据处理、`ETL` 任务以及大规模数据计算；`Trino` 则负责面向用户的交互式查询，提供统一的数据访问接口。 
在 `OpenLakes` 架构中，`Spark` 负责数据的加工与构建，将处理后的数据存储至 `Iceberg` 等数据湖存储系统中；`Trino` 通过 `Connector` 机制连接不同数据源，实现跨数据源的联邦查询，为用户提供统一的查询服务。
因此多模态查询应该选择在 `Trino` 查询框架基础上扩展多模态查询能力，实现对关系数据、图数据、向量数据以及时序数据的统一访问与查询。
- `Spark`用于接收传感器传入的数据, 从而进行处理, 最终存入数据库
- `Trino`用户用户端, 交互式查询

# 项目疑问
```
关系数据    → Trino支持
时序数据    → Trino支持, 但性能不是很好 (待定)
图数据      → Trino递归CTE (待定)
向量数据    → 不支持 (待定)
空间数据    → 需要插件（Trino有GeoSpatial插件, 可以支持空间数据）
```
## 1.是否需要使用时序数据库 & 图数据库
### 时序
`Trino`支持时序查询语法（时间函数、窗口函数、BETWEEN时间范围），基本的时间窗口查询已经验证可以跑通。但`Trino`本质上是一个通用查询引擎，不是专门的时序数据库，两者的差距主要体现在：
**存储层：** 专门的时序数据库（如`TDengine`, `InfluxDB`）针对时序数据做了列式压缩存储，同样的数据量存储空间可以少10倍以上。`Trino`用的是Parquet格式，压缩比不如时序专用格式。
**查询层：** 时序数据库内置降采样、插值、滑动窗口等时序专用操作，`Trino`需要手写复杂SQL实现同样的功能。
**性能临界点取决于采样间隔：**
- 如果传感器每小时采样一次，一个传感器一年只有8760条数据，几十个传感器也就几十万条，`Trino`完全够用
- 如果每分钟采样一次，一年就是52万条/传感器，100个传感器就是5200万条，这个量级下`Trino`查询延迟会明显上升
- 如果每秒采样，数据量级直接到亿级，必须考虑专门的时序数据库

### 图
需要根据图的规模来确定是否接入图数据库
## 2.是扩展`Trino`本身, 还是增加额外的协调层
扩展`Trino`内部,  还是在`Trino`外面加查询协调层
### 扩展`Trino`内部
- 给`Trino`增加新的查询引擎(时序/图/向量), 也就是扩充`Trino`的查询插件, 难度很大, 必须用Java，框架级开发，调试困难, 但性能更好
### 加查询协调层
- 添加一个新的查询协调层, 根据查询类型向下分配给`Trino`或其他数据库
- 难度中等, 架构清晰. 扩展性强, 接入新的数据库只需要添加新的查询分配模块
- 在需要扩充的数据库数量比较多的时候, 开发量更小
