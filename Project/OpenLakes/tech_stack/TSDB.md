#  核心概念                                                  
时序数据库（`Time Series Database`），是专为存储、查询和分析**时间序列数据**而设计的数据库系统所谓时间序列数据，是指每个**数据点都严格绑定一个时间戳**、并按时间顺序排列的数值或状态记录。打个比方：传统数据库（如MySQL）像一本通用百科全书，适合查某个人的完整信息；而时序数据库则像一本专设的《每日健康日志》，所有条目按日期自动装订成册，查“上周体温均值”只需抽出对应册子即可

时序数据与传统业务数据有显著差异:
- **写多读少**：数据持续高频写入，几乎从不修改或删除，属于“追加型”数据
- **时间驱动**：查询几乎总是围绕时间范围展开，如“过去24小时的平均温度”
- **体量巨大**：一个中等规模的物联网项目日增数据量可达数十亿条
- **价值随时间衰减**：最新数据最需要实时查询，历史数据更多用于趋势分析

## LSM-Tree 替代 B+ Tree
传统关系型数据库使用B+Tree作为存储结构，这在随机读写场景下表现良好。但时序数据是高频追加写入，`B+Tree`的随机写入会成为`I/O`瓶颈。时序数据库采用**LSM-Tree**（`Log-Structured Merge-Tree`）架构，通过内存缓冲+磁盘有序文件（`SSTable`）的组合，将随机写转换为顺序写，写入吞吐大大提升

### 列式存储
时序数据库采用**列式存储**而非行式存储——将同一指标在不同时间点的数据连续存储在一起。这种设计的优势在于：
- **极致压缩**：利用时序数据相邻点变化幅度小的特点，压缩率可达90%以上
- **按需读取**：查询时只需加载特定指标列，大幅减少I/O开销

## 时间分区
时序数据库会自动按时间（如每天、每小时）将数据切分成多个物理分片。查询“最近3天数据”时，系统只加载对应的几个分片，跳过其余数据，大幅提升响应速度。同时，过期数据只需直接删除对应分片即可，无需复杂的行级删除操作


# 应用场景
| 领域            | 典型场景                              |
| ------------- | --------------------------------- |
| **物联网与工业互联网** | 采集监控工厂设备运行状态（温度、压力、转速）、楼宇能耗、车辆轨迹等 |
| **电力能源**      | 智能电表每15分钟上报用电量，支撑实时负荷预测与异常识别      |
| **金融交易**      | 存储逐笔成交与行情快照，支持高频交易分析              |
| **IT运维监控**    | 采集服务器CPU、内存、磁盘等指标，实现系统性能监控        |
| **智慧交通**      | 采集车辆轨迹数据，支撑交通流量分析与智能调度            |
- 适合需要随着时间持续追加信息的业务场景
