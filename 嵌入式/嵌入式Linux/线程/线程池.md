**线程池是一种线程管理设计模式**。简单来说，它不是在需要执行任务时才创建一个新线程，而是在程序启动时预先创建好一批线程，并将它们“池化”起来。当有任务需要执行时，从池中取出一个空闲线程来运行；任务完成后，线程不销毁，而是放回池中等待下一个任务。
- 这能显著减少频繁创建和销毁线程的开销，尤其适合嵌入式或高性能服务中大量短暂任务的场景。

# 线程池的优势
|问题|没有线程池（每次新建）|有线程池|
|---|---|---|
|**创建/销毁开销**|每次任务都要创建和销毁线程（耗时、耗资源）|线程复用，仅创建一次，开销小|
|**资源控制**|线程数量不可控，可能耗尽系统资源|线程数量固定，资源可控|
|**响应速度**|创建线程需要时间，任务响应慢|线程已存在，任务可以立即执行|
|**管理复杂度**|需要手动管理线程生命周期|由线程池统一管理，简化代码|

在嵌入式环境中，这个开销更加明显。因为嵌入式系统的资源（内存、CPU）有限，反复创建线程（尤其是需要分配栈空间）是一项昂贵的操作。

# 工作原理
```
1. 初始化阶段：
   ├── 创建 N 个工作线程
   └── 所有线程开始等待任务

2. 任务提交阶段：
   ├── 调用者向线程池提交任务（函数指针 + 参数）
   ├── 任务被加入任务队列（加锁保护）
   └── 条件变量通知一个等待的工作线程

3. 任务执行阶段：
   ├── 工作线程从队列中取出任务（加锁保护）
   ├── 执行任务函数
   └── 完成后，线程回到等待状态

4. 销毁阶段：
   ├── 标记线程池为“正在销毁”
   ├── 唤醒所有等待的工作线程
   └── 等待所有线程结束，释放资源
```
