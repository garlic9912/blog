# 系统调用的差异性
每个操作系统都定义了自己独特的系统调用接口：
- **Linux**：提供了约**380个**系统调用，如 `fork()`, `open()`, `read()`, `write()`, `mmap()` 等。
- **Windows**：有自己的API（`Win32 API`），如 `CreateProcess()`, `OpenFile()`, `ReadFile()`，其底层系统调用机制（如 `sysenter` 指令）与Linux完全不同。

## 代码移植性差
当在做嵌入式Linux开发时，面对的**系统调用就是Linux那一套**。但如果以后需要将代码移植到其他`RTOS`或裸机环境，就需要特别注意了：

| 环境                   | 调用方式                                           | 你需要注意的                              |
| -------------------- | ---------------------------------------------- | ----------------------------------- |
| **嵌入式Linux**         | 使用标准的Linux系统调用（通过`glibc`或直接`syscall()`）。       | 只需熟悉Linux手册。                        |
| **裸机/RTOS**          | 通常没有系统调用，直接操作寄存器或调用RTOS提供的API（如`xTaskCreate`）。 | 原有的Linux系统调用代码需要**重写**，这称为“**移植**”。 |
| **Windows IoT Core** | 使用Win32 API。                                   | 需要学习一套全新的API。                       |


# `POSIX`
`POSIX`是`IEEE`（电气与电子工程师协会）制定的一套标准，它规定了操作系统应该提供哪些接口（系统调用、`Shell`命令等），让软件开发者能写出可以轻松移植到不同操作系统上的代码。
- **应用层**：如果你写的嵌入式Linux应用程序严格遵循`POSIX`标准，那么在更换硬件平台时，**应用层代码几乎不需要改动**。驱动程序才需要根据硬件适配。
- **底层/驱动层**：驱动开发本身不直接使用`POSIX`（它运行在内核态），但驱动的对外接口（如`read`, `write`系统调用）就是`POSIX`的一部分。
- **`RTOS`的“`POSIX`兼容层”**：很多高端`RTOS`（如`VxWorks, Zephyr`）会提供一套`POSIX`兼容的API，让你能把之前跑在`Linux`上的代码，相对容易地移植到`RTOS`上。


# 预处理
`gcc -E` 指示 `GCC` **只执行预处理阶段**，处理完源文件中的所有预处理指令（如 `#include`, `#define`, `#ifdef` 等）后就停止，**不进行编译、汇编和链接**.

## 预处理阶段的任务
- **头文件展开**：将 `#include` 指令替换为对应头文件的完整内容（递归展开）。
- **宏替换**：将 `#define` 定义的宏展开为对应的值或表达式。
- **条件编译处理**：根据 `#ifdef`, `#ifndef`, `#if`, `#else`, `#endif` 等指令，决定保留或丢弃哪些代码块。
- **注释删除**：移除所有 `//` 和 `/* */` 注释。
- **添加行号标记**：插入 `#line` 指令，用于后续编译阶段定位原始文件的行号。
- **处理特殊指令**：如 `#pragma`, `#error` 等。

## `.i`文件
- `test.c`文件
```c
#include <stdio.h>
#define PI 3.14159
int main() {
    printf("%f\n", PI);
    return 0;
}
```
- `test.i`文件
```c
# 1 "test.c"
# 1 "<built-in>"
# 1 "<command-line>"
# 1 "test.c"
# 1 "/usr/include/stdio.h" 1 3 4
# ...（这里展开 stdio.h 的全部内容，约 800 行）...
extern int printf (const char *__restrict __format, ...);
# ...（继续展开）...
# 3 "test.c" 2

int main() {
    printf("%f\n", 3.14159);
    return 0;
}
```

|特征|说明|
|---|---|
|**大量行号标记**|`# 1 "test.c"` 这类行，告诉编译器原始文件名和行号|
|**头文件完全展开**|所有 `#include` 被替换成头文件的完整代码|
|**宏被替换**|`PI` 变成了 `3.14159`|
|**注释消失**|`//` 和 `/* */` 全部删除|
|**条件编译消失**|`#ifdef` 等指令处理完，只剩被选中的代码|
|**没有 `#include`/`#define`**|所有以 `#` 开头的预处理指令都消失了（除了行号标记）|


# 链接
默认是动态链接
## 静态链接
将所有目标文件和所需要的库在编译时一并打包进最终的可执行文件. 库的代码被复制到最终的可执行文件中. 因此可执行文件在运行时不需要查找或加载外部库.

`-static`: 该参数指示编译器进行静态链接. 因为编译器默认的是动态链接, 只有显示指明这个参数才会进行静态链接
### 优点
1. **独立运行**：不需要依赖外部库文件，可以直接拷贝到其他机器运行
2. **启动快**：无需在运行时加载和解析共享库
### 缺点
1. **文件体积大**：每个程序都包含一份库的副本（如`libc.a`约`1MB+`）
2. **浪费内存**：多个进程同时运行时，库代码被重复加载
3. **更新困难**：库有安全漏洞时，需要重新编译所有依赖它的程序

## 动态链接
库在运行时才会被加载, 可执行文件仅包含了需要加载的库的路径和符号信息, 而不会包含全部代码, 也就是说库代码不包含在可执行文件中
### 优点
1. **节省空间**：多个进程共享同一份库文件（在内存中只加载一次）
2. **更新方便**：库更新时只需替换`.so`文件，无需重新编译程序
3. **按需加载**：程序运行时才决定加载哪些库
### 缺点
1. **依赖风险**：目标机器缺少对应库时程序无法运行（"找不到`.so`"）
2. **运行稍慢**：运行时需要加载和符号解析
