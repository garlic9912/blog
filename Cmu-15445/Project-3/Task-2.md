# # Inner Join & Left Join

两者在逻辑实现上的区别是: 在完全未匹配内表全部元组时是否构造NULL值来进行连接. 因此在代码实现上, 区别仅仅是一个`if`语句的判断, 可以复用绝大部分代码

# 难点在于如何保存状态

- 第一版: 缓存超过 `batch_size` 大小的元组数据 (`Result Buffering`)
    在 `Next` 时将产生的所有匹配结果存入一个临时队列（如 `std::deque`），若超过
    `batch_size` 则截断并留在下次使用。
  
  * 优点:
    
    * 逻辑简单直观: 只需要一个简单的队列来暂存溢出的元组。
    * 产出对齐: 能够严格保证每次向上层返回的 `Batch` 数量等于
      `batch_size`。
  
  * 缺点:
    
    * 内存压力极大: 如果 `Join` 产生笛卡尔积或大量匹配，缓存会瞬间膨胀，违背了批处理节省内存的初衷。
    * 控制流断层: 该方案只解决了“结果去哪”的问题，没有解决“如何跨`Next`
      调用保存子节点扫描进度”的问题。

- 第二版: 内存中开辟两个`batch_size`大小的缓冲区及游标 (`Input Buffering`)
  维护`left_buffer` 和 `right_buffer` 以及对应的两个游标 `left_cursor` 和
  `right_cursor`。
  
  - 优点:
    
    - 内存占用恒定: 内存开销被严格限制在
      `O(batch_size)`，不会随着匹配结果的数量增加而膨胀，非常安全。
    
    - 状态恢复自然: 游标和 `Buffer` 本身就是“执行现场”，在处理跨 `Batch` 的
      `Nested Loop` 逻辑时，状态转换非常清晰。

# 易错点

在每个算子的`Init()`函数里面一定要重置成员数据结构的状态, 否则可能会出现结果不断累加的情况

- 例如: 在执行聚合操作时, 未在`Init()`中充值哈希表的状态, 导致`count(*)`操作不断累加, 结果一直不断包含上次的状态
